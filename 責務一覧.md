# 責務一覧

本ドキュメントは `docs/.vitepress/config-builder` 周辺の機能と責務を1ページで整理したものです。

## 目的

- VitePress Docs Hub の設定処理を、変更しやすく・壊しにくく保つ。
- `型定義` / `データ取得` / `表示モデル変換` / `設定合成` の境界を明確にする。

## 機能一覧（現状）

1. プロジェクトディレクトリ検出
- 対象: `projectDiscovery.ts`
- 機能: `docs/project-docs/*` 配下の候補ディレクトリを列挙。

2. frontmatter 読み取り
- 対象: `projectFrontmatter.ts`
- 機能: 各プロジェクト `index.md` から `title/category/description` を取得。

3. submodule リポジトリ情報読み取り
- 対象: `projectRepository.ts`
- 機能: `.gitmodules` と git origin から `repoUrl` を解決。

4. プロジェクトモデル生成（正規化）
- 対象: `projectCatalog.ts`
- 機能: 検出結果 + frontmatter + repository を `DocProject[]` に統合。

5. バリデーション
- 対象: `validators.ts`
- 機能: `DocProject` の `name/path` を検証。

6. ナビゲーション生成
- 対象: `navBuilder.ts`
- 機能: `DocProject[]` から `projects/repositories` ドロップダウンを生成。

7. ホームサイドバー生成
- 対象: `homeSidebarBuilder.ts`
- 機能: docs 直下ページ + プロジェクトカテゴリ一覧をサイドバー化。

8. VitePress 設定合成
- 対象: `docs/.vitepress/config.mts`
- 機能: 上記の結果を `defineConfig` に合成して最終設定を生成。

## 責務境界（採用方針）

1. 型定義層
- 対象: `types.ts`
- 責務: 契約（型）だけを定義する。
- 非責務: I/O、バリデーション、変換ロジック。

2. データ取得層
- 対象: `projectDiscovery.ts`, `projectFrontmatter.ts`, `projectRepository.ts`
- 責務: ファイルシステムや設定ファイルからの読み取り。
- 非責務: UI 向け整形、VitePress 型への直接変換。

3. 正規化層
- 対象: `projectCatalog.ts`
- 責務: 入力データを `DocProject` に統合し、アプリ内で扱う共通モデルを作る。
- 非責務: Nav/Sidebar の UI 形状への変換。

4. 表示モデル変換層
- 対象: `navBuilder.ts`, `homeSidebarBuilder.ts`
- 責務: `DocProject[]` を VitePress テーマ型へ変換。
- 非責務: `.gitmodules` 解析や frontmatter 読み取り。

5. 設定合成層（Composition Root）
- 対象: `config.mts`
- 責務: 各層の呼び出し順と最終合成のみ。
- 非責務: 個別ルールの実装。

## 変更時の判断基準

1. その変更理由は1つか（SRP）。
2. そのファイルは1種類の依存に閉じているか（I/O or 変換 or 合成）。
3. テストをファイル単位で書けるか。

## 命名の見直し候補（任意）

- `projectRepository.ts` は実態が submodule 解析中心のため、将来的に `projectSubmoduleRepository.ts` などへ変更余地あり。
- `homeSidebarBuilder.ts` は実態が root/sidebar 全体寄りのため、将来的に `rootSidebarBuilder.ts` などへ変更余地あり。

## 補足

- `loadProjects` 内で `projectsRoot = join(docsRoot, PROJECT_DOCS_DIR)` を組み立てる方針は妥当。
- 理由: 呼び出し側の重複を減らし、`PROJECT_DOCS_DIR` の変更影響を局所化できるため。
